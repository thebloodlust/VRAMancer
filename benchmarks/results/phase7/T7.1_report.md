# T7.1 — Prompt lookup decoding (n-gram speculation, sans draft model)

Branch: `phase7/T7.1-prompt-lookup`.

## Hypothèse

En génération/édition de code, le modèle recopie souvent de longs n-grams
déjà présents dans le prompt (signatures de fonctions, blocs de code
inchangés, imports). `transformers` expose `prompt_lookup_num_tokens`
(`PromptLookupCandidateGenerator`) : à chaque étape, il cherche dans le
prompt un n-gram correspondant aux derniers tokens générés, propose les
tokens suivants comme candidats, et le modèle les **vérifie** en un seul
forward batché — par construction, lossless (les tokens rejetés sont
recalculés normalement).

## Implémentation

- `core/env_flags.py` : nouvelle propriété `PROMPT_LOOKUP` →
  `VRM_PROMPT_LOOKUP` (défaut `0`, doc table mise à jour).
- `core/backends.py`, `HuggingFaceBackend.generate()` Path 1 (single-GPU/CPU
  ou modèle `accelerate device_map="auto"`, seul chemin utilisant
  `model.generate()`) : si `VRM_PROMPT_LOOKUP=N > 0`, ajoute
  `gen_kwargs["prompt_lookup_num_tokens"] = N`. No-op par défaut
  (`VRM_PROMPT_LOOKUP=0`), donc aucun changement de comportement nominal —
  conforme à la règle de non-régression Phase 7 #2.
- Path 2 (pipeline bloc-par-bloc custom de VRAMancer, n'utilise pas
  `model.generate()`) : non concerné, le flag y est ignoré.

## Baseline / Mesure

`benchmarks/bench_prompt_lookup.py` (nouveau script) : 3 prompts différents
(édition de code), chacun construit à partir des ~200 premières lignes d'un
fichier source réel de VRAMancer + une consigne de modification répétitive
(ajout de docstrings, renommage, ajout de type hints). 256 tokens, greedy
(`do_sample=False`), `Qwen/Qwen2.5-7B-Instruct`, 1 GPU (RTX 3090).

Pour libérer la VRAM, le serveur de production (Qwen3.6-35B-A3B, port 5031) a
été arrêté temporairement le temps du bench, puis redémarré à l'identique.

| Prompt | Source | tok/s OFF (`VRM_PROMPT_LOOKUP=0`) | tok/s ON (`=10`) | gain |
|---|---|---|---|---|
| 0 | `core/inference_pipeline.py` (docstrings) | 24.64 | 104.33 | **+323%** |
| 1 | `core/backends.py` (rename `self.model`→`self._model`) | 26.39 | 158.65 | **+501%** |
| 2 | `core/api/registry.py` (type hints) | 26.48 | 175.51 | **+563%** |

Seuil d'acceptation : ≥20% de gain. **Mesuré : +323% à +563%** (3 prompts
différents, tous largement au-dessus du seuil).

## Vérification "lossless"

Comparaison texte brute (`pipe.generate()`) :

| Prompt | sorties identiques |
|---|---|
| 0 | non (OFF=1115 chars, ON=1141 chars, ON préfixe-extension de OFF) |
| 1 | non (OFF=991 chars, ON=1001 chars, idem) |
| 2 | oui (1282==1282) |

Ces écarts ont été root-causés via un diagnostic dédié
(`/tmp/diag_lookup_tokens.py`, comparaison directe des **token ids** retournés
par `model.generate()`, hors pipeline VRAMancer) sur le prompt 0 :

```
OFF n_tokens: 256   eos_at_end: False
ON  n_tokens: 261   eos_at_end: False
first_diff_token_idx: None of 256
identical_up_to_min_len: True
```

**Les 256 premiers tokens sont strictement identiques bit-à-bit** entre
`VRM_PROMPT_LOOKUP=0` et `=10` — le mécanisme est bien lossless comme annoncé
par `transformers`. L'écart observé au niveau texte vient du fait que
`PromptLookupCandidateGenerator` vérifie les candidats par **lot** : quand le
budget `max_new_tokens` est atteint au milieu d'un lot de candidats acceptés,
`model.generate()` peut renvoyer **quelques tokens de plus** que
`max_new_tokens` (ici 261 au lieu de 256, soit ~26 caractères — exactement la
longueur de l'extra dans la sortie texte). C'est un effet de bord cosmétique
de troncature, pas une divergence de contenu : le préfixe partagé est
identique token pour token.

Le prompt 2 n'a pas overshoot (sortie identique) — l'overshoot ne se produit
que lorsque le dernier lot de candidats accepté chevauche la limite
`max_new_tokens`.

## Quality non-regression check (règle Phase 7 #2)

- **Comparaison greedy token-à-token** : faite ci-dessus, identique sur les
  256 premiers tokens (cf. diagnostic) → pas de régression de contenu.
- **Perplexité wikitext-2** : non pertinente pour ce changement —
  `prompt_lookup_num_tokens` modifie uniquement la **stratégie de décodage**
  (quels tokens sont proposés puis vérifiés), pas le calcul des logits/scores
  du modèle. Une mesure de perplexité (teacher-forcing, sans génération)
  serait par construction strictement identique avec ou sans le flag — donc
  non exécutée (règle d'honnêteté : ne pas produire un chiffre qui ne mesure
  rien).
- **Tests de régression** : `tests/test_backend_stubs.py` +
  `tests/test_auto_detect_backend.py` → 12 passed (comportement par défaut
  `VRM_PROMPT_LOOKUP=0` non affecté).

## Tentative Qwen2.5-14B / 2 GPUs (à la demande de l'architecte)

La spec recommandait Qwen2.5-7B **puis 14B 2-GPU**. Tentative effectuée
(serveur de production re-arrêté/redémarré pour libérer la VRAM, comme pour
le run 7B) :

1. **bf16** (`device_map="auto"`, 2 GPUs, GPU0=23.6GB / GPU1=15.5GB) :
   `torch.OutOfMemoryError` pendant `caching_allocator_warmup` — GPU0 a
   absorbé ~20.6GB avant l'OOM, le fallback automatique de
   `core/inference_pipeline.py` (rechargement NF4) tente alors un 2e
   chargement **sans que la mémoire du premier essai ait été libérée**, et
   OOM immédiatement aussi.
2. **NF4 direct** (`VRM_QUANTIZATION=nf4`, pour éviter le double-chargement
   bf16→NF4 ci-dessus) : OOM différent, plus tard dans le chargement
   (`_finalize_model_loading` / `_move_missing_keys_from_meta_to_device`) —
   GPU0 se retrouve à 23.40/23.56 GB utilisés alors que GPU1 (15.5GB) reste
   largement sous-utilisé. Le `device_map` calculé pour le chargement NF4 ne
   répartit pas correctement le modèle 14B entre les deux GPU sur cette
   version de `transformers`/`bitsandbytes` — quasiment tout finit sur GPU0.

**Ceci n'est pas un problème lié à T7.1** : c'est un problème de répartition
`device_map` pré-existant du backend HF en mode NF4 multi-GPU (probablement à
investiguer séparément — sujet potentiel pour une future tâche Phase 7/8).
Conformément à la règle Phase 7 #6 (ne pas modifier le code/les scripts pour
faire passer un résultat), aucune correction n'a été tentée dans le cadre de
T7.1. Logs complets : `benchmarks/results/phase7/T7.1_14b_attempt_oom.{json,md}`.

Le serveur de production a été redémarré et confirmé `pipeline_loaded: true`
après cette tentative.

## Limites / non testé

- Le gain mesuré sur 7B (+323% à +563%) dépasse déjà le seuil d'acceptation
  (+20%) par un facteur 15-25×, sur 3 prompts de nature différente — le
  verdict SUCCÈS est donc solidement établi indépendamment de la tentative
  14B ci-dessus (bloquée par un problème d'infrastructure distinct).
- Path 2 (pipeline bloc-par-bloc custom, utilisé par le backend llama.cpp en
  production) ne bénéficie pas de ce flag — il n'utilise pas
  `model.generate()`. Hors scope T7.1 (qui cible le backend HF).

## Verdict

**SUCCÈS.**

- Gain tok/s mesuré : **+323% à +563%** sur 3 prompts de codage différents
  (seuil ≥20%).
- Lossless confirmé au niveau token (256 premiers tokens bit-identiques).
- Comportement par défaut (`VRM_PROMPT_LOOKUP=0`) inchangé, tests de
  régression OK.
- `VRM_PROMPT_LOOKUP=10` recommandé comme nouveau défaut pour le backend HF
  Path 1 en usage agent de codage (gain massif, pas de coût mémoire
  additionnel, pas de draft model à charger).
