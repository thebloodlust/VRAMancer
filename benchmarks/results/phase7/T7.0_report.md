# T7.0 — Prompt/prefix caching pour les agents de codage

Branch: `phase7/T7.6-autoheal` (pas de changement de code pour T7.0 — voir
verdict ; pas de nouvelle branche créée).

## Hypothèse

Les agents de codage renvoient le même long préfixe (system prompt +
contexte repo) à chaque appel ; réutiliser le KV cache du préfixe réduirait
massivement le TTFT de la 2e requête.

## Étape 1 — Backend llama.cpp : le cache de prompt est-il actif ?

`llama-cpp-python==0.3.22` (confirmé via `pip show`). Lecture du code source
(`llama_cpp/llama.py`, `Llama.generate()`) : il existe un mécanisme
**implicite** de réutilisation du KV cache par préfixe commun, actif par
défaut, sans flag ni configuration côté VRAMancer — à chaque appel
`Llama.generate(tokens, reset=True)` (chemin par défaut de
`_create_completion`, utilisé par `backend.generate()` /
`generate_stream()`), le code compare `tokens` aux `self._input_ids` de
l'appel précédent, calcule le préfixe commun le plus long, et tente
`self._ctx.kv_cache_seq_rm(-1, longest_prefix, -1)` pour ne ré-évaluer que le
suffixe divergent.

**Donc, en théorie, aucun changement de code n'était nécessaire** — le
backend llama.cpp de VRAMancer garde une seule instance `Llama` persistante
pour la durée du process serveur, condition suffisante pour que ce mécanisme
s'applique entre deux requêtes HTTP successives.

## Baseline / Mesure

`benchmarks/bench_prefix_cache_llamacpp.py` (nouveau script) — contre le
serveur live (Qwen3.6-35B-A3B, llama.cpp, port 5031) :

1. req1 : préfixe ~3262 tokens (extrait de `core/inference_pipeline.py`) +
   suffixe A → réchauffe le cache.
2. req2 "WITH cache" : même préfixe + suffixe B (TASK B au lieu de TASK A,
   ~10 derniers tokens différents) → mesure TTFT.
3. req éviction : préfixe différent (`core/backends.py`) + suffixe A → casse
   le préfixe commun.
4. req2 "WITHOUT cache" : même prompt qu'à l'étape 2, mesure TTFT.

| Metric | WITH cache | WITHOUT cache |
|---|---|---|
| TTFT (s) | 0.762 | 0.763 |
| total (s) | 1.350 | 1.349 |

**TTFT reduction mesurée : 0.2%** (seuil d'acceptation : ≥ 50%). Sorties
greedy identiques dans les deux cas (`outputs_identical=True`).

## Root cause (diagnostic, code annexe `/tmp/diag_prefix_cache.py`)

Le résultat ci-dessus était surprenant compte tenu du mécanisme décrit à
l'étape 1. Pour root-causer, le serveur de production a été **arrêté
temporairement** (~3 min, redémarré ensuite à l'identique — `pipeline_loaded:
true` confirmé après coup) et le même test a été rejoué directement sur
`llama_cpp.Llama(..., verbose=True)` pour voir les logs internes :

```
>>> req1: prompt_tokens=3262, total=1.433s
Llama.generate: 3248 prefix-match found but partial kv removal not supported, re-evaluating full prompt
>>> req2_with_cache: prompt_tokens=3259, total=0.773s
Llama.generate: 1 prefix-match found but partial kv removal not supported, re-evaluating full prompt
>>> evict: prompt_tokens=3272, total=0.788s
Llama.generate: 1 prefix-match found but partial kv removal not supported, re-evaluating full prompt
>>> req2_without_cache: prompt_tokens=3259, total=0.772s
```

**Root cause confirmé** : `kv_cache_seq_rm(-1, longest_prefix, -1)` retourne
`False` pour ce modèle — un préfixe commun de **3248/3262 tokens est bien
détecté**, mais la suppression partielle du KV cache n'est **pas supportée**,
donc llama.cpp ré-évalue systématiquement le prompt entier.

Cause architecturale probable : le GGUF de Qwen3.6-35B-A3B contient des
tenseurs `ssm_*` (`ssm_conv1d`, `ssm_dt`, `ssm_a`, `ssm_alpha`,
`ssm_beta`, `ssm_norm`, `ssm_out`) — c'est une architecture **hybride
attention + SSM/Mamba** (cf. familles Jamba/Qwen3-Next). Pour les couches
SSM, l'état récurrent à la position N dépend de tout l'historique 0..N de
façon non trivialement "tronquable" — `kv_cache_seq_rm` sur une plage
arbitraire n'est supportée par llama.cpp que pour les couches d'attention
classiques (KV cache pur), pas pour l'état récurrent SSM. D'où le fallback
"re-evaluating full prompt" qu'on observe pour CE modèle, alors que le même
mécanisme fonctionnerait probablement sur un modèle Transformer pur (ex.
Qwen2.5-14B en GGUF).

Note : malgré la ré-évaluation complète, le prefill de ~3260 tokens ne coûte
que ~0.77s sur ce hardware (MoE à ~3B paramètres actifs) — donc même un cache
fonctionnel n'aurait fait gagner qu'une fraction de ce ~0.77s de TTFT, pas un
facteur énorme, sur ce modèle précis.

## Étape 2 — Backend HF : `past_key_values` reuse

**Non implémenté**, conformément à la règle d'arrêt de T7.0 ("si
l'implémentation HF dépasse 2 jours, ne livrer que la voie llama.cpp") : la
réutilisation manuelle de `past_key_values` avec troncature au point de
divergence des token ids est une tâche de plusieurs jours (gestion des
positions, masques d'attention, et compatibilité avec le split multi-GPU
`accelerate`/`device_map`). De plus, le modèle de production actuel
(Qwen3.6-35B-A3B) utilise le backend llama.cpp, pas HF — la voie HF ne sert
pas Phase 6 dans l'immédiat.

## Verdict

**ÉCHEC** (documenté, root-caused — "un échec propre et documenté est un
résultat valide" per règle Phase 7 #4).

- Le mécanisme de cache de préfixe existe bien dans llama-cpp-python 0.3.22
  et **détecte correctement** un préfixe commun de 3248/3262 tokens.
- Mais il est **inopérant pour Qwen3.6-35B-A3B** car `kv_cache_seq_rm` ne
  supporte pas la suppression partielle sur ce modèle hybride
  attention+SSM (`partial kv removal not supported`).
- Gain TTFT mesuré : **0.2%**, très loin du seuil de 50%.
- Voie HF (`past_key_values` manuel) non implémentée, conformément à la
  règle d'arrêt.

**Implication pour la suite** : ce mécanisme pourrait fonctionner sur un
modèle Transformer pur (sans couches SSM) servi via llama.cpp — pas testé ici
(pas de petit GGUF Transformer-pur disponible localement sans perturber le
service). Si l'architecte souhaite explorer cette piste pour un futur modèle
non-hybride, le script `benchmarks/bench_prefix_cache_llamacpp.py` est
réutilisable tel quel.

## Quality non-regression check

Non applicable : aucun changement de code livré pour T7.0 (le mécanisme
testé est déjà natif à `llama-cpp-python`, et s'est révélé inopérant pour ce
