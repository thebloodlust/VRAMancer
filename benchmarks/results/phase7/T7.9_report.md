# T7.9 — Entrelacement de deux requêtes dans le pipeline (step 1 : pré-mesure de la bulle)

Branch: `phase7/T7.9-pipeline-interleave`. Script de mesure :
`benchmarks/probe_pipeline_bubble.py`.

## Hypothèse

En pipeline 2-GPU avec une requête unique, chaque GPU est oisif pendant que
l'autre calcule (~50% d'oisiveté) ; traiter deux requêtes décalées d'un étage
double presque le débit agrégé sans modifier le modèle.

## Étape 1 — Pré-mesure de la bulle (gate avant implémentation)

### Méthode

- `Qwen/Qwen2.5-7B-Instruct` chargé avec `device_map=None` (pas d'accelerate),
  ce qui fait emprunter à `split_model()` le chemin de découpage manuel
  VRAMancer ("Path 2", `self.blocks` avec `len==2`, `KVCacheBlock` par GPU) —
  confirmé : `split_model -> 2 blocks, devices=[1, 0], components=yes`,
  découpage 17/11 couches.
- `HuggingFaceBackend.infer()` instrumenté : chaque `block.forward()` est
  enveloppé de `torch.cuda.Event` (start/end + `synchronize(dev)`) pour
  mesurer le temps de calcul GPU pur de chaque bloc.
- 50 steps de décodage greedy (1 requête), prompt fixe
  `"Write a Python function that parses a CSV file and returns a dict."`.
- `idleness_i = 1 - busy_time_i / wall_time`.

### Résultat

| GPU | busy (ms) | busy % | idle % |
|---|---|---|---|
| cuda:0 | 709.2 | 31.3% | **68.7%** |
| cuda:1 | 1183.9 | 52.2% | **47.8%** |

Wall time total : 2266.1 ms pour 50 tokens (22.06 tok/s, 1 requête).

### Verdict step 1

**BULLE_SIGNIFICATIVE.** Les deux GPU sont oisifs une majorité du temps
(47.8% et 68.7%, seuil de la spec = 30%) — bien au-dessus du seuil sous
lequel "la tâche rapporterait peu". Le potentiel théorique d'entrelacement
est donc réel : avec 2 requêtes interfolées idéalement, le débit agrégé
pourrait approcher `busy_max / wall_time ≈` ne pas doubler exactement (les
deux blocs ont des charges différentes : 17 vs 11 couches → 52.2% vs 31.3%
busy), mais un gain de l'ordre de +40-60% de débit agrégé semble plausible
si l'entrelacement est parfait (limité par le GPU le plus chargé, cuda:1 à
52.2% busy → facteur ~1.9x théorique max, avant overhead de scheduling/KV
cache supplémentaire).

## Étape 2 — Implémentation du scheduler 2-slots

**NON FAITE À CE STADE** — conformément à la procédure de validation
tâche-par-tâche, ce rapport documente uniquement l'étape 1 (pré-mesure,
qui gate la suite). L'étape 2 (ordonnanceur à 2 slots dans le serveur,
`VRM_PIPELINE_SLOTS=2`, synchronisation par CUDA events sans locks Python
autour des forwards, KV caches séparés par requête) est une implémentation
substantielle touchant le serveur de production et nécessite une validation
explicite avant de démarrer, étant donné :

- la complexité (ordonnancement entrelacé bloc-par-bloc entre 2 requêtes,
  gestion de 2 KV caches concurrents) ;
- le risque (règle d'arrêt de la tâche : "si après 1 semaine les sorties sont
  corrompues par interférence des caches/streams, fermer et documenter — signe
  que l'architecture actuelle ne s'y prête pas sans refonte") ;
- le fait que "Path 2" (découpage manuel sans accelerate) n'est PAS le chemin
  utilisé en production actuellement (le serveur tourne en llama.cpp ; le
  backend HF en production utilise `device_map="auto"` → Path 1/accelerate,
  cf. T7.1). Le scheduler T7.9 ciblerait donc Path 2, qui devrait d'abord être
  amené en production pour bénéficier de T7.9 — **point à clarifier avec
  l'architecte avant d'investir dans l'étape 2**.
